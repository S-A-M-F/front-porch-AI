# Rawhide — What's New (Nightlies)

These notes feed the in-app "Update Available" dialog for Rawhide / cutting-edge builds.

## Recent improvements

- 🔄 **Home screen refresh button** — Added a refresh icon next to the multi-select button on the character grid. Lets users instantly pick up new characters added by external tools like Character Card Forge without restarting the app.

- 🐛 **Fixation and realism baseline no longer bleed between chats** — Starting a new chat with a character (or tapping a brand-new card on the home grid with 0 prior sessions, or opening a freshly imported card) now always begins with a clean emotional state. Fixation thoughts, arousal, bond, trust, emotion, and time-of-day no longer carry over from the previous character or prior session. "Reading the room" retroactive evaluations now trigger correctly for new conversations. (Rawhide)

- 🖼️ **Draw Things local image generation (macOS) is now reliable** — The previous hacky temp-script bridge has been replaced with a proper bundled PyInstaller JSON CLI sidecar (`dt_grpc_client`) reusing the battle-tested Python gRPC+FlatBuffer+NNC protocol. End users have zero Python/grpc dependency. Core plumbing, storage (dedicated host/port + 8 rich DT parameters), service wiring, reference images (img2img), and web remote control are complete and working. The primary settings dialog shows the correct gRPC instructions + dedicated host/port fields + DT-native sampler list + Advanced subsection for Shift/Strength/TeaCache/CFG Zero Star (wired to the new storage when the Draw Things backend is selected). Full 19-sampler exposure and advanced controls are now user-facing for DT. Test + Generate work from all call sites. (Rawhide)

- 🛡️ **Very large chats are now much more resilient to old corrupted data** — Chats with thousands of messages (especially those that have been heavily forked, regenerated, or used with the Realism Engine for a long time) could accumulate invalid per-message state in the database (e.g. bad `swipeIndex` values). Previously this would cause hard crashes with a solid red error overlay in debug builds, making the chat unusable until you manually exported/reimported. The app now defensively sanitizes this state on load, clamps indices safely in the model, and gracefully handles bad historical data so the chat loads cleanly. Export/reimport still works as a nuclear reset, but you shouldn't need it just to recover a chat anymore.

- 🛡️ **Group Card export/import now guarantees 100% member fidelity — no partial groups ever** — Even characters without avatars (or whose private avatar file is missing at export time) are always fully included when exporting a group as a .group.png. A complete placeholder PNG with the full V2 character data + metadata embedded is synthesized instantly for any such member, so the exported card contains exactly N usable members for an N-member group. Import materializes real private avatars for all of them; "Separate to my library" (extract) works on every single one for the recipient. Zero silent drops, zero warnings, zero compromises — shared groups are now reliably complete and splittable for everyone.

- 📊 **Home screen message counts and filters now actually work** — The per-character "user messages" count chips and the "Recent activity" / "Messages sent" sort options on the home grid were showing 0 or useless data for a long time (a regression from the group chat overhaul). Fixed the key mismatch between how the database caches were populated (using internal dbId) and how the UI looked them up (using the stable filename-based ID). Both the chips and the two data-dependent filters now reflect real usage again and survive app restarts.

- 🖼️ **Characters with no avatars now work properly in groups** — Previously, adding a library character that had no avatar image (or whose file was missing) to a group could result in that member being invisible or dropped in the chat sidebar and home grid. The group creation and "Add to group" paths, plus the always-on schema repair, now guarantee every private group member gets a valid avatar PNG (with full metadata embedded) even if the source had none. You can safely use any character in groups.

- 🛠️ **Group Card export/import round-trips are now more robust and user-friendly** — Exporting a group now shows a clear warning snackbar (and logs details) if any members lack usable avatars instead of silently producing a partial card. Partial imports (some members succeed, others fail) now use much clearer messaging that explicitly explains the outcome and reassures that successful members remain fully usable in the created group. The DB schema repair safety net now also covers the `group_members` table for any future column evolution using the same PRAGMA/ALTER/backup pattern as other tables. Added automated round-trip tests exercising realism baseline + defaultMemberRealismState (incl. relationships), per-character objectives, character system prompt overrides, and multi-member fidelity through the PNG export/import path.

- 🛡️ **Group schema safety improvements for long-lived databases** — The automatic repair that runs on every launch now better protects users with years of chat history when the new decoupled `group_members` table is involved: it creates the table with the modern contract (including `created_at`), emits a clear warning if any orphaned group member rows are detected, and has improved on-disk comments to help external tools understand the private-avatar rules. First mutation still creates a timestamped backup.

- 🐛 **Fixed launch crash on macOS (and other platforms)** — The app would fail to build/run with "Can't find '}' to match '{'" errors in the character card grid. This was caused by an incomplete async loader for the new decoupled group cards (massive planning comments + duplicate code fragments left unbalanced braces). Group home-grid cards now properly load real member counts and avatar collages using FutureBuilder + StorageService.groupsDir (matching the pattern used in chat and edit flows). The app launches cleanly again.

- 🛡️ **Group characters fully decoupled from the singular library (clean break, no blobs, no shared IDs)** — Group members are now first-class separate entities stored in a new normalized `group_members` Drift table (typed columns for all card data + single primary avatarFilename; UUID internal IDs per member). Private avatar PNGs live exclusively under `groups/<groupId>/avatars/` (StorageService). "Add from library" in creator/edit copies the card + PNG at that instant into the group's private storage (never mutates library). Group Card import populates *only* the private group storage + table rows — the library is never touched (no opt-in, no side effects, "never allow" automatic paths). The *only* bridge to your library is the explicit "Separate to my library" button. Multi-avatar/expressions not supported in groups. No migration (feature never shipped). Old `GroupChat.characterIds`, all global-repo resolution for groups, and shared-file hazards are deleted. Safety: deleting/editing a library character can never affect groups, and vice versa. (Rawhide 2026-05)

- ✏️ **Edit Group now uses the same tabbed editor style as 1:1 character editing** — Right-click "Edit Group" (home grid group cards or GroupMemberCard in active group chat sidebar) opens a dedicated tabbed EditGroupPage matching EditCharacterPage visuals (section cards, AppColors-only, TabBar in AppBar). Personality & World pruned (explanatory note + Group System Prompt + Scenario + Per-Character Overrides only; no Description/Personality). Dialogue keeps First Message + disabled "Coming soon" stub for Alternate Greetings; Example Dialogue fully removed. Creation wizard untouched. Old edit paths + dead edit code in CreateGroupChatPage cleaned up.

- 🛠️ **Group Chat Creator is now production-ready** — A long-standing ID resolution bug has been fixed that caused groups created in the new wizard to appear with "0 characters" on the home screen after the first use (even though they worked the first time you entered the chat). Character identity resolution is now consistent across the group creator, chat service, and home screen, so groups created via the wizard reliably show their members and load correctly every time.

  The wizard has also been restructured so the scenario and first message step now happens **after** Group Dynamics. AI-generated openings can now intelligently incorporate the hidden inter-character relationships you pre-seeded. The generation prompts themselves have been significantly strengthened with clear explanations of the relationship scale and behavioral guidance, so the resulting scenes actually reflect the dynamics between the characters.

- 🛠️ **Group realism & needs settings now actually stick** — The entire group chat creation → first entry → settings stack for the Realism Engine and Needs simulation has been audited and fixed. Settings from the creator (master toggle + per-member seeds + Needs on/off) now correctly promote the active flags in ChatService on first entry, so the engine and UI indicators work immediately instead of being silently off. The Realism & Needs tab in Group Settings was also completely dead (update handlers were empty stubs; status/dirty scaffolding was unused). The toggles now live-apply via the proper service setters. You can finally turn realism on for groups, whether at creation time or later in settings, and it persists.

- 🐛 **Custom group first messages now load correctly on New Chat** — "New Chat" flows (session picker, in-chat menu, and immediately after group creation) were silently using the first character's default firstMessage instead of a custom group-level first message the user had set in the creator or imported via Group Card. The priority logic is now identical and correct in both places that plant a fresh opening message. The previous fragile ensure workaround has been removed entirely.

- 🛡️ **Group characters now fully decoupled from the singular library (clean break)** — See the detailed entry earlier in this file. Import/create/edit now use private group_members rows + groups/<id>/avatars/ only. The sole bridge to library is explicit "Separate to my library". No more shared files/IDs or automatic pollution. (Fix round completed member persistence in high-risk paths using generalized duplicate + typed inserts.)

- 🛡️ **Your years of character chats are now safe from silent schema problems** — Long-time users (especially those with large databases full of group chats, objectives, Chaos/Realism/Needs history) will no longer hit "no such column" crashes or broken group features after app updates. The database layer now performs an automatic, non-destructive repair on every launch: it inspects the real columns with PRAGMA, adds any that a past migration silently missed (chat_id scoping, all the v30-v32 group columns for baseline realism, Chaos toggles, lore/world scoping, per-character system prompts, etc.), and — the first time it actually changes anything — creates a timestamped backup copy of your .db right next to the original. You can always rename the backup back if you ever want to roll back. Deleting or resetting your database is never required or suggested. Group chat parity features (per-character objectives, Chaos sidebar, full Group Card roundtrips, etc.) now work reliably even on the oldest surviving user databases.

- 🛠️ **Group Chat creator now supports per-character baseline realism + group-scoped system prompts at creation time** — The Create Group dialog has been overhauled with full per-character editors for baseline Bond/Trust/Emotion/Arousal/Needs/Fixations (matching the live realism engine) and dedicated group-scoped system prompt text fields for each member. These seed the initial state exactly like the Group Settings dialog, so groups start with the intended personality overrides and realism baselines instead of defaults. Both creator and settings paths now use the same v30-backed persistence for group_realism_state and per-character group prompt overrides.

- 🧠 **Group realism & needs are now properly stored in the database (no more hidden jank)** — Per-character Bond/Trust/Emotion/Fixation/Arousal/Relationships + full Needs simulation state for every member in a group chat now lives in the new `sessions.group_realism_state` and `groups.default_member_realism_state` columns (schema v30, clean break). This was the last piece needed for high-fidelity Group Cards: exported `.group.png` files now carry the current evolved state for all members. When you split a group member out to a solo character, they keep the relationships, needs, and fixation they developed inside the group. Import round-trips the state perfectly into the new group definition. The old `__group_state__` sentinel message system is fully removed.

- 📚 **Baked-in Lorebooks now export correctly in character PNGs** — The "Lorebook" tab in the character editor (the keyword-triggered world info entries that travel with the card) is now properly included when you Export PNG from the grid or via the browser UI. Previously the web export path and an avatar-replacement code path silently dropped the `character_book` data, so imported cards in SillyTavern, Risu, Chub, etc. would be missing their embedded lore. Both desktop and web export paths now go through the canonical CharacterCard serializer and emit full ST-compatible lorebook entries (with keys, insertion_order, position, etc.). Round-trips are now reliable.

- 🧹 **Internal code hygiene improvements** — Began introducing barrel files (`models/models.dart`, `services/services.dart`, `utils/utils.dart`, `ui/widgets/widgets.dart`) to dramatically reduce the hundreds of repetitive import lines that had accumulated. New and refactored code now carries far less boilerplate. This is an ongoing, opportunistic cleanup (we convert files when we touch them for real work) rather than a single giant effort. No user-visible behavior changes; this makes future development faster and the codebase easier to maintain.

- ⚙️ **Auto-Configure for local models is now actually accurate** — The "Auto-Configure" button (in both the chat "Model" dialog and main Settings) finally produces GPU layer recommendations that make sense when you have a specific context size in mind. It now uses real model architecture data from the GGUF file (layer count, KV head dimensions) plus a proper binary search to find the highest number of layers that should safely fit your chosen context + KV quantization without OOMing. The old "ratio * 40" magic math and "100 MB per 1k context" guesswork are gone for the important path. Works dramatically better with the VRAM override for testing tight scenarios.

- 🔧 **Model Settings dialog cleanups** — The "Model" sheet (Model button in any chat) no longer shows a spurious API Key input when the 🍎 oMLX backend is active. The global "Request Reasoning" controls, the local "Thinking Model" switch, and the four hardware acceleration chips (Use Vulkan / ROCm / CuBLAS / Metal) have all been removed from this dialog. These advanced options remain available in the main app Settings and via .kcpps presets / Auto-Configure.

- 🧠 **Needs chips now survive app restart** — The Energy / Hunger / Hygiene / Bladder / etc. delta chips under the last AI message are now persisted the same way the classic Bond / Trust / Emotion chips have always been. Closing and reopening the app no longer wipes the needs row while leaving the realism chips intact.

- 🖋️ **Chat composer: live red spell underlines + stable custom coloring** — The message input now reliably shows both at the same time. Flutter's built-in spell check pipeline is bypassed entirely (it refused to call our custom `buildTextSpan()` when spell results were available). Instead, the desktop spell service runs independently with a 300ms debounce, and the `StyledTextController` merges wavy red underlines onto the correct spans inside its own `buildTextSpan()`, preserving each segment's amber ("dialogue") or blue (*action*) color. Right-click suggestions read from the controller's cache via the new `SpellCheckResultsProvider` interface. Layout, height, resize handle, and keyboard behavior are unchanged.

- 🧠 **Realism Engine speedup** — Removed duplicate code paths for KoboldCpp vs API backends. All realism evaluations (emotion, bond, trust, narrative) now run concurrently instead of sequentially, improving chat response times.


- 🍎 **oMLX backend support (macOS Apple Silicon)** — New backend option for local LLM inference via oMLX (https://github.com/jundot/omlx). Select oMLX in Settings → Backend, then use the model picker to choose from your loaded oMLX models. Also available as a per-chat model override in Chat Settings. Install oMLX via `brew install jundot/omlx/omlx` and run `omlx serve`.

- 🔧 **oMLX toggle in Chat Model Settings + compact buttons** — The "Model Settings" dialog (gear menu in any chat) now includes the 🍎 oMLX option on macOS, matching the main backend settings. Backend toggle buttons are also more compact (tighter padding and smaller text) to reclaim screen space.

- ⚡ **Faster KoboldCpp responses** — KoboldCpp has a built-in FIFO task scheduler that can batch multiple requests. The previous code used sequential `await` chains for realism evals and `waitForIdle()` before each eval, defeating KoboldCpp's built-in batching. Changed all 4 realism eval calls (relationship, emotional state, physical state, narrative) from sequential `await` to concurrent `Future.wait()`, and removed the `waitForIdle()` call before each eval. This improves chat response times by allowing KoboldCpp to batch multiple eval requests.

- 📺 **Token throttle defaults to OFF** — The "Smooth Output Buffer" (token throttle) now defaults to OFF for all users — new and existing. Existing stored preference is deleted on startup so users get the full speed. Tokens display at raw GPU speed instead of artificially slowed down. Users can re-enable via UI if they prefer the smoother display.

- 🧠 **Needs simulation fixes** — Three fixes for the Needs system:
  - **Needs reset on message delete** — Deleting a character message now properly restores the needs state from the previous message, not just the last message. Previously deleting a non-last character message would leave needs at their current (post-turn) values instead of rolling back.
  - **Hygiene no longer raises during sex** — Orgasm/sexual activity now properly lowers hygiene instead of potentially raising it. The daily activity check could incorrectly detect "bathing" during sex scenes and add +25 hygiene, overriding the -6 from climax. Fixed by gating the daily activity check behind cooldown and increasing the climax hygiene penalty.
  - **Needs chips show on regenerated messages** — Needs change chips now display when regenerating messages, not just on new messages. The needs_deltas were computed before `_generateResponse` ran, meaning the post-generation checks hadn't modified the needs vector yet. Moved the computation to after generation so chips show the actual delta.

- 🎯 **Objectives no longer bleed between chats** — Quest/objective goals were previously stored per-character, so objectives created in one chat would appear in another chat with the same character. Fixed by scoping objectives to the chat session (same mechanism used for messages). Each conversation now has its own independent set of objectives.

- 🎯 **Needs chips now render after regen** — Regenerating a message was stripping emotion_label from the metadata because the regen path never set it in `_pendingRealismMetadata` (the normal path does this after realism evals). Without emotion_label, the realism indicator early-return suppressed all chips. Fixed by adding the same `emotion_label` + `realism_state` synthesis into the regen path.
- 🛠️ **Group creator + Group Cards now fully reliable** — Fixed a critical ID resolution bug that caused newly created groups to open with "0 characters" and a broken UI. The group wizard and chat loading layer now consistently resolve character identities, so "Create Group & Enter Chat" works correctly every time (including with full Realism data). Group Cards with per-member realism/relationships are also more robust. The entire new group creation flow is now considered production-ready.

- 🔄 **Group wizard restructured** — Scenario and first message generation have been moved from early in the flow to the final steps (after Group Dynamics). AI-generated openings can now intelligently reflect the hidden inter-character relationships the user pre-seeded.
- ✨ **Much stronger AI opening generation for groups** — The prompts for generating scenario and first message now include rich, well-explained inter-character dynamics data (with scale guidance, tier meanings, and clear behavioral instructions). Generated openings are far more likely to naturally reflect pre-set tensions, affections, rivalries, etc. between group members.

- 🪄 **Group Lore Entry editor completely overhauled** — The "Add/Edit Group Lore Entry" dialog (used in the group creator and Group Settings) no longer has atrocious word-wrapping ("Enable d", "Consta nt"), dead switches, or a broken slider. Replaced the problematic layout with clean, non-wrapping rows, proper subtitles explaining what "Constant" actually does, live-updating toggles + slider via StatefulBuilder, and consistent AppColors card styling. Both the creator and live group settings now match the quality of the rest of the group UI.

- 🐛 **"Create Only" in the group creator no longer crashes on macOS** — The "Create Only (don't enter chat yet)" button on the final Review step was wired to the exact same code as "Create & Enter Chat". It performed a full `setActiveGroup` + `startNewChat` + route replacement anyway. On macOS this combination could hard-crash the app. The button now does a pure save + pop (no ChatService side effects, no forced navigation). The new group simply appears in your list on the home screen. The main "Create Group" button and the primary "Create & Enter" button continue to work exactly as before.
